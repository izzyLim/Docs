실전 워크플로우

가이드

"어떻게 셋업하는가"는 하네스 가이드에 있어요. 이 문서는 실제 작업 상황에서 어떤 순서로, 어떤 프롬프트로 Claude를 쓰는가를 다룹니다.

이 문서의 전제
CLAUDE.md가 세팅되어 있고, Claude Code CLI를 쓸 수 있는 상태입니다. 각 워크플로우는 독립적으로 읽을 수 있어요.

기본 기본 워크플로우

01. 새 프로젝트 시작

🚀

상황: 새 프로젝트를 Claude와 함께 처음 시작할 때

CLAUDE.md 작성 → 구조 설계 → 초기 코드 생성

1

CLAUDE.md 먼저 작성

프롬프트

아래 정보를 바탕으로 이 프로젝트의 CLAUDE.md를 작성해줘. 프로젝트: BIP-Pipeline 목적: 스마트폰 수요 예측 데이터 파이프라인 기술 스택: Python 3.11, FastAPI, Oracle 19c, Airflow 팀 규칙: snake_case, 타입 힌트 필수, docstring 필수 금지 사항: 프로덕션 DB 직접 접근, .env 출력

2

프로젝트 구조 설계

프롬프트

CLAUDE.md를 읽었으면, 이 프로젝트의 디렉토리 구조를 제안해줘. FastAPI 앱, Airflow DAG, Oracle 연동, 테스트 폴더 포함. 실제 생성하기 전에 구조만 먼저 보여줘.

3

구조 확인 후 생성 지시

프롬프트

구조가 맞아. 이제 생성해줘. 각 파일은 기본 보일러플레이트만 — 복잡한 로직은 아직 넣지 말고. 생성 완료 후 파일 목록 요약해줘.

💡

포인트 구조를 먼저 보고 확인한 다음 생성하세요. Claude가 한 번에 너무 많은 걸 만들면 검토하기 어려워요.

02. 기능 구현

⚙️

상황: 새 기능을 Claude에게 구현 요청할 때

요청 → 계획 확인 → 구현 → 테스트

1

구현 전 계획 먼저 확인

프롬프트 — 계획 요청

NL2SQL 엔드포인트를 구현하려고 해. 요구사항: - POST /api/v1/query - 자연어 입력 → Oracle SQL 생성 → 실행 → 결과 반환 - 쿼리 실행 타임아웃: 30초 - 결과 최대 1000행 코드 바로 작성하지 말고, 구현 계획을 먼저 보여줘. 어떤 파일을 수정하고 어떤 순서로 진행할지.

2

계획 확인 후 구현 지시

프롬프트 — 구현 지시

계획 좋아. 그대로 진행해줘. 단, SQL 생성 로직은 일단 TODO로 남겨두고 나머지 라우터, DB 연결, 응답 형식부터 완성해줘.

3

테스트 먼저, 그 다음 실행

프롬프트 — 테스트

구현된 엔드포인트의 pytest 테스트를 작성해줘. 정상 케이스 2개, 에러 케이스 3개 (타임아웃, 빈 결과, DB 연결 실패). 테스트 작성 후 실행해서 결과 알려줘.

03. 버그 디버깅

🔍

상황: 오류가 발생했는데 원인을 모를 때

오류 공유 → 원인 분석 → 수정 → 검증

기본 디버깅 프롬프트

아래 오류가 발생했어. 오류 메시지: [오류 전문 붙여넣기] 오류가 나는 코드: [관련 코드만 발췌] 실행 환경: Python 3.11, Oracle 19c, FastAPI 0.104 재현 조건: POST /api/v1/query에 한국어 자연어 입력 시 1. 원인 분석 2. 수정 방법 3. 같은 패턴의 다른 버그 있는지 확인 순서로 진행해줘. 수정 전에 분석 결과 먼저 보여줘.

원인을 못 찾을 때 — 단계적 접근

심화 디버깅 프롬프트

원인을 모르겠어. 단계적으로 디버깅해줘. 1단계: 가능한 원인 후보 목록 (가능성 높은 순) 2단계: 각 원인 확인하는 방법 3단계: 확인 코드 작성해서 실행 원인 찾으면 알려주고, 못 찾으면 더 좁혀나갈게.

⚠

Claude가 같은 수정을 반복할 때 동일한 오류에서 3번 이상 같은 수정을 제안하면 접근 방식을 바꿔야 해요. "이 방법은 이미 시도했어. 다른 관점으로 접근해줘"라고 명시하세요.

04. 코드 리뷰

👁️

상황: 작성한 코드를 Claude에게 검토 받을 때

코드 제출 → 리뷰 → 수정 적용

코드 리뷰 프롬프트

아래 코드를 리뷰해줘. 파일: nl2sql.py 변경 목적: Oracle NL2SQL 쿼리 생성 및 검증 특히 확인해주세요: SQL 인젝션 방지, 타임아웃 처리 [코드 붙여넣기] 리뷰 항목: 1. 버그 가능성 (있으면 라인 번호 포함) 2. 보안 취약점 3. 성능 이슈 4. 코드 컨벤션 위반 심각도 기준으로 정렬해서 보여줘. 수정 제안도 포함해줘.

PR 리뷰 요청 시

PR 리뷰 체크리스트 생성 요청

이 변경사항을 PR로 올릴 거야. PR 설명과 함께 검토자가 확인해야 할 체크리스트를 만들어줘. 변경 내용: - NL2SQL 엔드포인트 추가 - Oracle 연결 풀 설정 - 입력 검증 로직 추가 보안팀 리뷰가 필요한지, DB 검토가 필요한지도 판단해줘.

1

현황 분석 먼저

프롬프트

이 모듈을 리팩토링하고 싶어. 먼저 현재 문제점만 분석해줘. [코드 또는 파일명] 분석 항목: - 코드 중복 - 지나치게 긴 함수 - 의존성 문제 - 테스트 어려운 부분 아직 수정하지 말고 분석만.

2

우선순위 합의 후 점진적 적용

프롬프트

💡

    ├── status-backend.md        ← 백엔드 진행 상태 (자동 업데이트)
    ├── status-frontend.md
    └── status-data.md

12. NL2SQL 작업

🗄️

Oracle DB에서 NL2SQL 쿼리를 생성하고 검증할 때

메타데이터 확인 → SQL 생성 → 검증 → 실행

NL2SQL 요청 프롬프트 (SKILL.md 기반)

다음 질의에 대한 Oracle SQL을 생성해줘: "지난 분기 브랜드별 스마트폰 판매량 상위 10개 모델" nl2sql SKILL.md에 따라 진행해줘: 1. 관련 테이블 메타데이터 확인 (OpenMetadata 조회) 2. SQL 초안 생성 (Oracle 19c 문법) 3. 실행 계획 확인 4. 결과 샘플 검증 SQL 실행 전에 생성된 쿼리를 먼저 보여줘.

쿼리 검증 프롬프트

검증 프롬프트

이 SQL을 검증해줘. [SQL 붙여넣기] 확인 항목: - Oracle 19c 호환 문법인지 - 전체 테이블 스캔 없는지 (EXPLAIN PLAN) - WHERE 조건이 빠진 UPDATE/DELETE 없는지 - 인덱스 활용 가능한지

13. ML 모델 개발

🤖

모델 실험 프롬프트

스마트폰 주간 판매량 예측 모델을 개발하려고 해. 데이터: - 기간: 2022-2024 주별 판매량 - 피처: 브랜드, 모델명, 지역, 출시일, 마케팅 비용 요구사항: - 예측 기간: 12주 - 평가 지표: MAPE < 15% - 해석 가능성 필요 (SHAP) 1. 베이스라인 모델 먼저 (SimpleExpSmoothing) 2. 결과 확인 후 XGBoost나 Prophet으로 업그레이드 결정 베이스라인 코드 작성해줘.

모델 비교 프롬프트

XGBoost와 Prophet 모두 학습됐어. XGBoost MAPE: 12.3% Prophet MAPE: 14.8% 두 모델을 비교 분석해줘: - 어떤 상황에서 각 모델이 강점인지 - 앙상블이 의미 있을지 - 운영 관점 (재학습 주기, 유지보수 난이도) 최종 추천 모델과 이유도 알려줘.
