1. 도입이 필요한 상황 vs 불필요한 상황

LangGraph 필요

  • 에이전트가 결과를 보고 다시 시도해야 할 때 (루프)
  • 질문에 따라 다른 도구/경로를 선택해야 할 때 (분기)
  • 사람이 중간에 승인/검토해야 할 때
  • 장시간 실행 중 크래시 복구가 필요할 때
  • 여러 에이전트가 협업해야 할 때

LangGraph 불필요

  • 단순 질문 → LLM → 답변 (직선)
  • 문서 검색 → 요약 (RAG 파이프라인)
  • 정해진 순서대로 처리 (ETL)
  • → 이 경우 LangChain 또는 직접 구현이 더 단순

2. BIP 프로젝트 실전 사례

BIP-Pipeline에서 LangGraph를 사용하는 3개 에이전트와 그 구조를 소개합니다. 모두 Haiku + MCP 도구 기반의 ReAct 에이전트입니다.

from langgraph.prebuilt import create_react_agent
from langchain_mcp_adapters.client import MultiServerMCPClient

# MCP 도구 로드
mcp_client = MultiServerMCPClient({
    "bip-stock": {"url": "http://bip-stock-mcp:8000/sse", "transport": "sse"}
})
tools = await mcp_client.get_tools()

# ReAct 에이전트 생성
llm = ChatAnthropic(model="claude-haiku-4-5-20251001", max_tokens=4000)

# 실행
result = await agent.ainvoke({"messages": [{"role": "user", "content": user_message}]})

첫 에이전트: 가장 간단한 ReAct 패턴

from langgraph.prebuilt import create_react_agent
from langchain_anthropic import ChatAnthropic
from langchain_core.tools import tool

# 1. 도구 정의
@tool
def search_db(query: str) -> str:
    """데이터베이스에서 검색"""
    # 실제 DB 쿼리 로직
    return f"검색 결과: {query}"

@tool
def get_current_price(ticker: str) -> str:
    """종목 현재가 조회"""
    return f"{ticker}: 50,000원"

# 2. 에이전트 생성
llm = ChatAnthropic(model="claude-haiku-4-5-20251001")
agent = create_react_agent(
    llm,
    tools=[search_db, get_current_price],
    prompt="당신은 금융 데이터 분석 전문가입니다."
)

# 3. 실행
result = agent.invoke({
    "messages": [{"role": "user", "content": "삼성전자 현재가 알려줘"}]
})

Phase 2: 프로덕션 기반 구축

PostgreSQL 체크포인터 설정

from langgraph.checkpoint.postgres.aio import AsyncPostgresSaver

# 체크포인터 생성
checkpointer = AsyncPostgresSaver.from_conn_string(
    "postgresql://user:pass@localhost:5432/dbname"
)

# 에이전트에 적용
agent = create_react_agent(llm, tools, checkpointer=checkpointer)

# thread_id로 대화 지속
config = {"configurable": {"thread_id": "user-123-session-1"}}
result = await agent.ainvoke({"messages": [...]}, config=config)

# 같은 thread_id로 이전 대화 이어가기
result2 = await agent.ainvoke({"messages": [새 메시지]}, config=config)

Human-in-the-Loop 설정

# 특정 노드 전에 실행 중지
agent = create_react_agent(
    llm, tools,
    checkpointer=checkpointer,
    interrupt_before=["critical_action"]  # 이 노드 전에 멈춤
)

# 실행 → 중단점에서 멈춤
result = await agent.ainvoke(input, config)
# → 사용자에게 검토 요청 → 승인 후 재개
result = await agent.ainvoke(None, config)  # 이어서 실행

Phase 3: 멀티 에이전트 + 웹 UI

Supervisor 패턴 구현

from langgraph.graph import StateGraph, END

class SupervisorState(TypedDict):
    messages: Annotated[list, operator.add]
    next_agent: str

def supervisor(state):
    """질문을 분석하고 적절한 에이전트로 라우팅"""
    response = llm.invoke(...)
    return {"next_agent": response.content}

def data_agent(state):
    """DB 조회 전문"""
    ...

def analysis_agent(state):
    """분석 전문"""
    ...

# 그래프 구성
workflow = StateGraph(SupervisorState)
workflow.add_node("supervisor", supervisor)
workflow.add_node("data_agent", data_agent)
workflow.add_node("analysis_agent", analysis_agent)

workflow.set_entry_point("supervisor")
workflow.add_conditional_edges("supervisor", lambda s: s["next_agent"])
workflow.add_edge("data_agent", "supervisor")      # 완료 → supervisor로 복귀
workflow.add_edge("analysis_agent", "supervisor")

app = workflow.compile(checkpointer=checkpointer)

6. 운영 팁 & 트러블슈팅

흔한 문제

  • 에이전트 무한 루프 — 반드시 max iteration 설정. BIP에서는 iteration < 3
  • MCP 연결 끊김 — SSE 연결 타임아웃. try/finally로 mcp_client.close() 보장
  • 토큰 폭발 — 긴 대화는 context_compressor로 요약. 또는 컨텍스트 리셋
  • 체크포인터 스키마 변경 — State에 필드 추가 후 기존 체크포인트와 호환 안 됨. 마이그레이션 필요

성능 최적화

  • Haiku 우선 — 단순 판단은 Haiku, 복잡한 분석만 Sonnet/Opus
  • 도구 병렬 호출 — MCP 도구를 asyncio.gather로 병렬 실행
  • 캐싱 — 자주 조회하는 데이터는 Redis 캐시
  • 스트리밍 — invoke 대신 stream으로 사용자 대기시간 단축

BIP에서 얻은 교훈

교훈상세
프롬프트가 핵심 LangGraph 구조보다 시스템 프롬프트 품질이 결과를 결정. 방향성 규칙(환율 higher_is_risk), 출력 형식 강제, 금지 사항 등 명시적으로.
도구 선택은 LLM에 맡겨라 Phase 2에서 룰 엔진으로 도구를 강제 매핑했지만, Phase 5에서 LLM ReAct로 전환. 새 항목 대응이 훨씬 유연.
하네스는 임시방편 모델이 진화하면 이전 우회 로직은 기술 부채. Opus 4.6에서 스프린트 구조 불필요해진 것처럼.
테스트 환경 분리 테스트 DAG는 반드시 send_telegram=False. 운영 채널에 테스트 메시지 나가면 혼란.
감사 로그는 처음부터 나중에 추가하면 기존 호출 이력이 없음. agent_audit_log를 에이전트 구축과 동시에.
